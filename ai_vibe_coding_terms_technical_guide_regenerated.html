基礎開發與版本控制

1. Repository / Repo｜儲存庫

專案的完整程式碼與文件空間。

工作原理：保存原始碼、設定、測試、文件與歷史版本。

應用場景：AI Agent 讀取專案結構、定位入口檔、決定修改範圍。

2. Commit｜提交

一次可追蹤的修改紀錄。

工作原理：Git 將檔案差異、作者、時間與說明記成版本快照。

應用場景：讓 AI 修改有還原點，也方便寫 changelog。

3. Branch｜分支

從主線分出的開發路線。

工作原理：不同分支可以平行修改，完成後再合併。

應用場景：讓 AI 在 feature/fix 分支工作，避免破壞 main。

4. Pull Request / PR｜合併請求

把分支修改送出審查。

工作原理：PR 集中呈現 diff、討論、檢查結果與合併紀錄。

應用場景：AI 完成修改後開 PR，交給人類或另一個 AI review。

5. Diff｜差異比較

修改前後的行級差異。

工作原理：顯示新增、刪除、異動的程式碼。

應用場景：檢查 AI 到底改了什麼，避免被摘要誤導。

Prompt 與上下文工程

6. Prompt｜提示詞

給 AI 的任務說明。

工作原理：模型根據 prompt、上下文與工具結果產生回應。

應用場景：描述需求、限制、輸出格式、測試方式。

7. System Prompt｜系統提示詞

高優先級的行為規則。

工作原理：設定角色、規範、語氣、邊界與工具使用策略。

應用場景：規定 AI 只能做最小修改、必須列出測試。

8. Context｜上下文

AI 當下可見的所有資訊。

工作原理：包含對話、程式碼、文件、錯誤訊息與工具回傳。

應用場景：提供完整上下文可大幅降低亂猜。

9. Context Window｜上下文視窗

模型一次能處理的文字容量。

工作原理：超過容量時需要摘要、切片或檢索。

應用場景：大型專案要分階段讀檔，不能全部塞進去。

10. Few-shot Prompting｜少樣本提示

提供範例讓 AI 模仿。

工作原理：模型從例子推斷格式與風格。

應用場景：固定 commit message、題庫 JSON、教案格式。

11. Context Engineering｜上下文工程

設計 AI 看見什麼、何時看、看多少。

工作原理：比單純 prompt 更重視資料選擇、壓縮、工具回傳與記憶。

應用場景：大型 Agent 要管理任務狀態、檔案摘要與檢索結果。

12. Prompt Template｜提示詞模板

可重複套用的 prompt 結構。

工作原理：保留角色、任務、限制、輸出格式，填入變數。

應用場景：建立「除錯模板」「重構模板」「文章模板」。

13. Instruction Hierarchy｜指令階層

不同來源指令的優先順序。

工作原理：系統、開發者、使用者、工具資料的權重不同。

應用場景：防止外部 README 覆蓋使用者任務。

14. Role Prompting｜角色提示

指定 AI 扮演某種專家。

工作原理：角色會影響檢查角度與輸出風格。

應用場景：資深前端、資安審查員、自然科教材設計師。

15. Constraint｜限制條件

要求 AI 必須遵守的邊界。

工作原理：例如不可改資料庫、不可新增套件、不可重寫全文。

應用場景：AI Coding 最重要的安全護欄之一。

16. Output Format｜輸出格式

要求回答的固定格式。

工作原理：可指定表格、JSON、步驟、檔案清單。

應用場景：讓 AI 回報修改內容時更容易檢查。

17. Self-critique｜自我檢查

要求 AI 先檢查自己的方案。

工作原理：模型產出後再從風險、測試、遺漏角度檢討。

應用場景：提交前請 AI 列出可能破壞點。

18. Plan-then-Act｜先規劃後執行

先產生計畫，再修改檔案。

工作原理：降低 Agent 無目標亂改的風險。

應用場景：大型任務要求 AI 先列分步計畫。

19. Guardrail Prompt｜護欄提示

限制 AI 行為的提示規則。

工作原理：明定禁止事項、審核點與失敗時處理。

應用場景：禁止刪除檔案、禁止改 package lock。

LLM 與生成模型

20. LLM｜大型語言模型

能理解與生成文字、程式碼的模型。

工作原理：根據上下文預測高機率輸出。

應用場景：解釋錯誤、寫函式、補文件、重構程式。

21. Token｜詞元

模型處理文字的基本單位。

工作原理：文字會被切成 token，用於計算上下文與成本。

應用場景：估算長文件是否需要拆分。

22. Temperature｜溫度

控制輸出隨機程度。

工作原理：低溫較穩定，高溫較有創意但風險高。

應用場景：寫程式低溫；命名、文案可稍高。

23. Hallucination｜幻覺

AI 產生看似合理但錯誤的內容。

工作原理：模型不是資料庫，上下文不足時可能編造。

應用場景：要求 AI 列出實際檔案、測試與來源。

Agent 與自主工作流

24. Agent｜代理人

能規劃、使用工具並執行任務的 AI。

工作原理：結合模型、工具、記憶、環境與回饋迴圈。

應用場景：自動讀 repo、改檔、跑測試、開 PR。

25. Coding Agent｜程式代理

專門處理軟體開發任務的 Agent。

工作原理：可讀寫檔案、執行 shell、檢查測試結果。

應用場景：Codex、Claude Code、Copilot agent、Cursor agent。

26. Vibe Coding｜氛圍式寫程式

用自然語言驅動 AI 快速開發。

工作原理：人類決定方向與審查，AI 產生原型與迭代。

應用場景：教材網站、HTML 工具、小型 app 原型。

27. Orchestrator｜編排者

負責拆任務與分派工作的主控角色。

工作原理：把大任務拆成設計、實作、測試、文件。

應用場景：多模型協作或多 Agent 分工。

28. Human-in-the-loop｜人類在迴圈中

關鍵決策需要人類確認。

工作原理：AI 可建議與執行低風險步驟，高風險步驟需審核。

應用場景：部署、刪除資料庫、寄信、金鑰操作前確認。

29. Agent Loop｜代理迴圈

AI 反覆觀察、思考、行動、檢查。

工作原理：Observe → Plan → Act → Reflect → Repeat。

應用場景：長任務修 bug、跑測試、根據結果再修。

30. ReAct｜Reason + Act 模式

推理與行動交替進行的 Agent 方法。

工作原理：先判斷下一步，再呼叫工具，根據結果再決策。

應用場景：搜尋資料、讀檔、執行測試時常見。

31. Reflection｜反思步驟

AI 對產出進行二次檢查。

工作原理：檢查是否符合需求、是否漏測、是否有風險。

應用場景：修改完成後要求 AI 自評。

32. Task Decomposition｜任務拆解

把大任務拆成小步驟。

工作原理：每一步有輸入、輸出、檢查點。

應用場景：把「做網站」拆成架構、內容、樣式、互動、測試。

33. Sub-agent｜子代理

專門負責某類任務的 Agent。

工作原理：主代理分派給測試、文件、資安、前端子代理。

應用場景：多模型協作工作流。

34. Multi-agent｜多代理系統

多個 Agent 合作完成任務。

工作原理：不同角色互相審查、補位、分工。

應用場景：一個寫程式、一個審查、一個寫文件。

35. Memory｜記憶

AI 保存的長期或短期資訊。

工作原理：可保存偏好、專案規則、決策、進度。

應用場景：記住你的 HTML 技術文件風格與命名習慣。

36. Scratchpad｜暫存工作區

AI 用來整理中間資訊的空間。

工作原理：保存待辦、假設、檔案摘要、測試結果。

應用場景：長任務避免忘記前面已檢查的內容。

37. Checkpoint｜檢查點

任務中的可回復狀態。

工作原理：在重要修改前後保存狀態。

應用場景：大型 refactor 前先建立 git commit。

38. Autonomy Level｜自主程度

AI 可自行決策與執行的範圍。

工作原理：從只建議，到可改檔，到可部署，風險逐步升高。

應用場景：依任務風險設定審核門檻。

工具呼叫與結構化輸出

39. Function Calling｜函式呼叫

模型選擇並填寫函式參數。

工作原理：LLM 產生結構化呼叫，由外部程式執行。

應用場景：查日曆、建立檔案、呼叫 API。

40. Tool Calling｜工具呼叫

AI 使用外部工具完成任務。

工作原理：工具可包含搜尋、檔案、終端機、瀏覽器、資料庫。

應用場景：AI 跑 npm test、讀 PDF、查 GitHub issue。

41. Structured Output｜結構化輸出

要求 AI 產生固定格式。

工作原理：使用 JSON、表格、schema 降低解析錯誤。

應用場景：自動生成題庫、行事曆、設定檔。

42. JSON Schema｜JSON 結構規格

描述 JSON 欄位與型別的規則。

工作原理：可驗證必要欄位、格式、巢狀結構。

應用場景：讓 AI 產生可被程式穩定讀取的資料。

MCP 與外部系統

43. MCP｜Model Context Protocol

AI 應用連接外部資料與工具的開放標準。

工作原理：像 AI 的 USB-C，讓模型存取檔案、資料庫、工具與工作流。

應用場景：把 GitHub、Drive、資料庫、校本教材庫接到 AI。

44. MCP Server｜MCP 伺服器

提供工具與資源給 AI 的服務。

工作原理：宣告可用工具、資源與提示，讓 AI client 呼叫。

應用場景：建立自然科教材查詢 server。

RAG 與知識庫

45. RAG｜檢索增強生成

回答前先檢索資料再生成。

工作原理：查詢相關文件片段，交給 LLM 根據來源回答。

應用場景：打造教案、研習、科展資料知識庫。

46. Embedding｜嵌入向量

把文字轉成語意數字向量。

工作原理：語意相近的文字向量距離較近。

應用場景：搜尋「馬達不轉」也能找到「無法運作」。

47. Vector Store｜向量資料庫

儲存與搜尋 embedding 的資料庫。

工作原理：以相似度找出最相關片段。

應用場景：建立黑熊老師自然科學知識庫。

48. Chunk｜文件切片

把長文件切成小段。

工作原理：讓檢索能取回精準段落，而不是整份文件。

應用場景：PDF、教案、技術文件分段索引。

49. Retrieval｜檢索

根據問題找出相關資料。

工作原理：可用關鍵字、向量搜尋、混合搜尋與 metadata。

應用場景：查比賽、法規、教材段落。

50. Grounding｜依據來源回答

讓 AI 回答有可驗證根據。

工作原理：把輸出限制在來源內容與檢索結果上。

應用場景：技術文章、政策建議、期刊投稿需要引用。

測試、除錯與品質

51. Bug｜錯誤

程式沒有照預期運作。

工作原理：可能源自語法、邏輯、環境、資料或依賴。

應用場景：HTML 打不開、簡報亂碼、App 閃退。

52. Debug｜除錯

找出錯誤原因並修正。

工作原理：重現問題、讀錯誤、縮小範圍、修正、驗證。

應用場景：要求 AI 先列假設再做最小修改。

53. Stack Trace｜錯誤堆疊

錯誤發生時的呼叫路徑。

工作原理：顯示檔案、行號、函式與例外訊息。

應用場景：貼給 AI 可快速定位問題。

54. Unit Test｜單元測試

測試單一函式或模組。

工作原理：用小輸入驗證輸出是否正確。

應用場景：測日期轉換、表單驗證、資料解析。

55. Integration Test｜整合測試

測試多模組串接是否正常。

工作原理：檢查前端、後端、API、資料庫互動。

應用場景：表單送出後寫入 Supabase 並顯示成功。

56. Regression Test｜回歸測試

確認新修改沒有弄壞舊功能。

工作原理：每次修改後重跑既有測試。

應用場景：AI 改首頁後確認搜尋、複製、RWD 沒壞。

57. Eval｜模型評估

用固定任務評估 AI 表現。

工作原理：衡量正確性、格式、引用、安全與可執行性。

應用場景：建立 AI 產生教案或程式碼的評分集。

58. Lint｜靜態程式檢查

檢查語法、風格與潛在錯誤。

工作原理：不執行程式也能檢查程式碼問題。

應用場景：ESLint、Ruff、flake8。

59. Formatter｜格式化工具

自動整理程式排版。

工作原理：統一縮排、空格、換行與引號風格。

應用場景：Prettier、Black。

60. Type Checking｜型別檢查

檢查資料型別是否一致。

工作原理：在執行前發現函式參數、回傳型別錯誤。

應用場景：TypeScript、mypy、pyright。

61. Static Analysis｜靜態分析

不執行程式就分析風險。

工作原理：檢查安全漏洞、壞味道、死碼。

應用場景：AI review 後再跑掃描工具。

62. Smoke Test｜冒煙測試

快速確認基本功能能啟動。

工作原理：不用完整測試，只確認主要流程沒壞。

應用場景：部署後先檢查首頁、登入、主要按鈕。

63. E2E Test｜端對端測試

模擬使用者完整操作流程。

工作原理：從瀏覽器操作到後端資料完整驗證。

應用場景：Playwright/Cypress 測教材網站互動。

64. Test Fixture｜測試夾具

測試前準備的固定資料或環境。

工作原理：讓測試可重複、可預測。

應用場景：建立假使用者、假題庫、假回應。

65. Mock｜模擬物件

用假的依賴替代真服務。

工作原理：避免測試時真的呼叫 API 或資料庫。

應用場景：模擬 OpenAI API 回傳。

66. Assertion｜斷言

測試中用來判斷結果的條件。

工作原理：如果結果不符合預期，測試失敗。

應用場景：expect(total).toBe(100)。

67. Coverage｜測試覆蓋率

測試涵蓋程式碼的比例。

工作原理：可看出哪些檔案或分支沒被測到。

應用場景：AI 補測試時參考 coverage 報告。

68. Flaky Test｜不穩定測試

有時通過有時失敗的測試。

工作原理：常由時間、網路、非同步、外部依賴造成。

應用場景：AI 修 CI 失敗時要辨識是否 flaky。

69. Snapshot Test｜快照測試

保存輸出快照並比較差異。

工作原理：UI 或序列化輸出變動時會提醒。

應用場景：檢查名詞卡片 HTML 是否意外改變。

部署、環境與自動化

70. CI｜持續整合

每次提交後自動建置與測試。

工作原理：由 workflow 在 push 或 PR 時執行測試。

應用場景：AI 開 PR 後自動跑 lint/test/build。

71. CD｜持續部署

測試通過後自動部署。

工作原理：將合併後的程式發布到正式或測試環境。

應用場景：GitHub Pages、Netlify、Vercel 自動部署。

72. Workflow｜工作流程

自動化任務設定。

工作原理：由事件觸發 jobs，例如 push、PR、排程。

應用場景：GitHub Actions YAML。

73. Docker｜容器化工具

封裝應用與依賴的工具。

工作原理：用 image/container 建立一致環境。

應用場景：避免「我電腦可以跑，你電腦不行」。

74. Container｜容器

隔離執行的應用環境。

工作原理：容器包含執行所需檔案與依賴，彼此隔離。

應用場景：讓 AI 在容器中跑測試，降低主機風險。

75. Environment｜環境

程式執行的情境。

工作原理：常分 dev、staging、production。

應用場景：AI 修 bug 要確認錯誤在哪個環境發生。

76. Staging｜預備環境

正式上線前的測試環境。

工作原理：接近 production 但不影響真使用者。

應用場景：AI 修改先部署 staging 測試。

77. Production｜正式環境

真使用者正在使用的環境。

工作原理：穩定性、安全性與備份要求最高。

應用場景：AI 不應直接改 production。

78. Rollback｜回滾

把系統恢復到先前版本。

工作原理：部署失敗時快速切回穩定版本。

應用場景：AI 改壞網站時用 git 或平台還原。

79. Release｜發布版本

把功能正式交付使用。

工作原理：可包含版本號、release notes、tag。

應用場景：v1.2.0 新增搜尋與一鍵複製。

80. Semantic Versioning｜語意化版本

用 MAJOR.MINOR.PATCH 表示版本。

工作原理：破壞性變更、功能新增、修補分別升版。

應用場景：套件與教材工具版本管理。

81. Feature Flag｜功能旗標

用開關控制功能是否啟用。

工作原理：不重新部署也能開關新功能。

應用場景：先讓教師帳號試用新 AI 功能。

82. Observability｜可觀測性

了解系統內部狀態的能力。

工作原理：結合 log、metrics、traces。

應用場景：AI 修線上問題時要先看監控資料。

83. Monitoring｜監控

持續追蹤系統狀態。

工作原理：檢查錯誤率、回應時間、CPU、流量。

應用場景：網站掛掉時自動警示。

84. Incident｜事故

影響服務的異常事件。

工作原理：需要記錄時間線、影響、原因與修復。

應用場景：AI 部署造成網站不可用。

安全與風險

85. Sandbox｜沙盒

受限制的執行環境。

工作原理：限制檔案、網路、命令與權限。

應用場景：AI Agent 改程式時避免誤刪或執行危險指令。

86. Prompt Injection｜提示注入

把惡意指令藏在資料中誘導 AI。

工作原理：AI 可能把外部資料誤當指令執行。

應用場景：讀 README、網頁、email 前要有安全邊界。

87. Data Leakage｜資料外洩

敏感資料被不該看到的系統取得。

工作原理：可能經 prompt、log、API、工具回傳外流。

應用場景：學生個資、API key 不可貼入不可信工具。

88. Secret｜機密金鑰

API key、token、密碼等。

工作原理：應放環境變數或 secret manager，不可寫死。

應用場景：OPENAI_API_KEY、SUPABASE_SERVICE_ROLE_KEY。

89. Permission｜權限

AI 或工具可做的操作範圍。

工作原理：最小權限原則能降低失控影響。

應用場景：允許讀檔與測試，但不直接部署或刪庫。

90. Supply Chain Risk｜供應鏈風險

第三方套件與工具帶來的風險。

工作原理：npm 套件、GitHub Action、MCP server 可能被污染。

應用場景：AI 建議安裝套件前要查來源與權限。

91. Indirect Prompt Injection｜間接提示注入

惡意指令藏在外部資料中。

工作原理：AI 讀網頁、文件、README 時被資料操控。

應用場景：Coding Agent 讀不可信 repo 特別危險。

92. Capability｜能力範圍

工具或 Agent 被授權可做的事。

工作原理：包含讀檔、寫檔、網路、執行命令、發送請求。

應用場景：MCP server 應清楚宣告與限制能力。

93. Least Privilege｜最小權限

只給完成任務必需的權限。

工作原理：即使 AI 被誘導，也降低傷害。

應用場景：只開專案資料夾，不開整個硬碟。

94. Approval Gate｜審核閘門

高風險操作前的人工確認點。

工作原理：刪除、部署、寄送、付費、外部請求需確認。

應用場景：AI 自動化流程必備。

95. Allowlist｜允許清單

只允許指定項目通過。

工作原理：比黑名單更容易控管風險。

應用場景：只允許 AI 執行 npm test、npm run build。

96. Denylist｜拒絕清單

禁止特定命令或資源。

工作原理：阻擋 rm -rf、curl pipe shell 等危險模式。

應用場景：AI 終端機安全策略。

97. Provenance｜來源脈絡

資料或決策從哪裡來。

工作原理：追蹤來源可判斷可信度與責任。

應用場景：AI 引用文件、工具回傳、外部網頁時標示來源。

98. Audit Log｜稽核紀錄

記錄誰在何時做了什麼。

工作原理：用於追蹤安全事件與錯誤操作。

應用場景：AI Agent 操作檔案、API、部署都要留紀錄。

99. PII｜個人可識別資訊

可識別個人的資料。

工作原理：姓名、email、電話、學生資料都可能是 PII。

應用場景：餵 AI 前要遮蔽學生姓名與聯絡資訊。

100. Threat Model｜威脅模型

系統可能遭遇的攻擊與風險分析。

工作原理：列出資產、攻擊者、入口、影響與防護。

應用場景：設計 AI Agent 前先想誰能誘導它做壞事。

工程工作流

101. Refactor｜重構

不改功能但改善結構。

工作原理：整理命名、拆模組、移除重複、降低耦合。

應用場景：AI 很適合做小步重構，但要測試保護。

102. Boilerplate｜樣板程式碼

專案常見起始架構。

工作原理：包含初始化設定、目錄、範例元件。

應用場景：快速建立 React、Express、HTML 工具。

103. Scaffold｜鷹架

建立專案骨架。

工作原理：先生成資料夾、路由、元件與設定檔。

應用場景：讓 AI 先搭自然科互動教材網站骨架。

104. Code Review｜程式碼審查

檢查程式正確性、安全與維護性。

工作原理：審查 diff、架構、測試、風險與風格。

應用場景：AI 改完後請另一個 AI 或人類 review。

105. Technical Debt｜技術債

快速開發留下的結構問題。

工作原理：短期省時間，長期增加維護成本。

應用場景：Vibe Coding 連續補丁容易累積技術債。

106. Minimal Reproducible Example｜最小可重現範例

能重現 bug 的最小程式。

工作原理：排除無關因素，只保留錯誤核心條件。

應用場景：向 AI 提問時提供最小範例，答案更準。

需求與產品定義

107. User Story｜使用者故事

用使用者角度描述需求。

工作原理：通常寫成：身為某角色，我想做某事，以便達成某目標。

應用場景：把模糊需求轉成 AI 可拆解的功能。

108. Acceptance Criteria｜驗收條件

判斷功能是否完成的標準。

工作原理：用可測試條件定義完成狀態。

應用場景：避免 AI 只做出看起來像完成的原型。

109. Specification / Spec｜規格書

功能、限制、資料、流程的明確描述。

工作原理：把需求從聊天語言轉成工程規格。

應用場景：交給 AI 前先寫 spec，能降低返工。

110. PRD｜產品需求文件

產品層級的需求說明。

工作原理：包含目標、使用者、功能、限制、成功指標。

應用場景：大型 AI 專案開始前先產生 PRD。

111. Scope｜範圍

這次任務要做與不做的邊界。

工作原理：明確界定檔案、功能、時間與風險。

應用場景：要求 AI 不碰登入、不改資料庫、不重寫樣式。

112. Out of Scope｜不在範圍內

明確排除的項目。

工作原理：避免 AI 順手多做造成破壞。

應用場景：本次只修匯入，不重新設計 UI。

113. MVP｜最小可行產品

能驗證核心價值的最小版本。

工作原理：先做最少功能確認方向，再逐步擴充。

應用場景：用 AI 快速做教學工具原型。

114. Spike｜技術探勘

短時間研究可行性的任務。

工作原理：先驗證方案、套件、API 是否能用。

應用場景：請 AI 比較三種 PDF 轉 HTML 方法。

115. Backlog｜待辦清單

尚未完成的需求集合。

工作原理：可依優先級、難度、風險排序。

應用場景：AI 可幫你把聊天需求整理成 backlog。

116. Roadmap｜路線圖

專案功能演進計畫。

工作原理：分階段安排版本、里程碑與依賴。

應用場景：把教材平台規劃成 v1/v2/v3。

架構與設計

117. Architecture｜系統架構

系統元件與關係的整體設計。

工作原理：決定前端、後端、資料庫、API 與部署方式。

應用場景：讓 AI 先畫架構再寫程式。

118. System Design｜系統設計

從需求推導技術方案。

工作原理：考慮資料流、容量、安全、可靠性與維護性。

應用場景：避免 AI 只產生單檔式玩具程式。

119. Design Doc｜設計文件

描述技術決策與替代方案。

工作原理：記錄為什麼這樣設計、取捨與風險。

應用場景：讓 AI 提案前先產生 design doc。

120. ADR｜Architecture Decision Record

架構決策紀錄。

工作原理：用短文件記錄某次重要技術選擇。

應用場景：為什麼選 Supabase 而不是 Firebase。

121. Module｜模組

可獨立理解與維護的程式單元。

工作原理：把功能拆成邊界清楚的檔案或套件。

應用場景：AI 重構時將搜尋、匯出、UI 拆模組。

122. Component｜元件

可重複使用的 UI 或功能單位。

工作原理：前端框架中常用 props/state 組合畫面。

應用場景：按鈕、名詞卡、搜尋框、複製區塊。

123. API｜應用程式介面

系統之間交換資料的約定。

工作原理：定義路徑、方法、參數、回傳與錯誤。

應用場景：AI 連接前端與後端時必須先確認 API。

124. Endpoint｜端點

API 的具體網址或方法。

工作原理：例如 GET /terms、POST /feedback。

應用場景：除錯時確認 AI 呼叫的 endpoint 是否正確。

125. Payload｜資料負載

請求或回應中傳送的資料。

工作原理：通常是 JSON body、表單或檔案。

應用場景：AI 產生 API 測試時要檢查 payload 格式。

126. State｜狀態

程式當下記錄的資料。

工作原理：UI、使用者、表單、載入狀態都可能是 state。

應用場景：互動教材中目前題目、分數、進度。

127. Stateless｜無狀態

每次請求不依賴伺服器記憶。

工作原理：狀態放在 client、token 或資料庫。

應用場景：API 設計、serverless function 常見。

128. Stateful｜有狀態

系統會保存前後關係。

工作原理：session、連線、長任務進度都屬於狀態。

應用場景：聊天機器人、任務佇列、WebSocket。

工具與終端機

129. CLI｜命令列介面

用文字指令操作工具。

工作原理：透過 terminal 執行安裝、測試、建置、部署。

應用場景：Codex CLI、Git、npm、Docker。

130. Shell Command｜終端機指令

在作業系統中執行的命令。

工作原理：可讀寫檔案、安裝套件、啟動服務。

應用場景：AI 執行前要審查高風險命令。

131. Working Directory｜工作目錄

目前指令執行所在資料夾。

工作原理：相對路徑會依工作目錄解讀。

應用場景：很多 AI 失誤來自跑錯資料夾。

132. Environment Variable｜環境變數

系統層級的設定值。

工作原理：程式可從環境讀取 key、mode、URL。

應用場景：存 OPENAI_API_KEY、DATABASE_URL。

133. Package Manager｜套件管理器

安裝與管理依賴的工具。

工作原理：npm、pnpm、pip、uv、poetry 等。

應用場景：AI 新增套件前要確認必要性。

134. Dependency｜依賴套件

專案需要的外部程式庫。

工作原理：記錄在 package.json、requirements.txt 等。

應用場景：更新依賴可能修 bug，也可能造成破壞。

135. Lockfile｜鎖定檔

固定依賴版本的檔案。

工作原理：確保不同環境安裝相同版本。

應用場景：package-lock.json、pnpm-lock.yaml、poetry.lock。

136. Build｜建置

把原始碼轉成可執行或可部署成果。

工作原理：編譯、打包、壓縮、產生靜態檔。

應用場景：npm run build、vite build。

137. Dev Server｜開發伺服器

本機預覽與熱更新服務。

工作原理：監看檔案變更並即時刷新。

應用場景：AI 修改前端後用 localhost 檢查。

138. Log｜日誌

程式執行紀錄。

工作原理：記錄錯誤、請求、狀態與診斷資訊。

應用場景：貼 log 給 AI 可快速除錯。

資料、API 與後端

139. Database｜資料庫

保存結構化資料的系統。

工作原理：用表格、文件、鍵值或圖結構儲存。

應用場景：使用者、文章、回饋、檢核表資料。

140. Schema｜資料結構

資料庫或資料格式的欄位規則。

工作原理：定義欄位、型別、關聯與限制。

應用場景：Supabase 表格設計。

141. Migration｜資料庫遷移

資料庫結構的版本化變更。

工作原理：用腳本新增欄位、表格、索引。

應用場景：AI 改資料庫前必須產生 migration。

142. ORM｜物件關聯映射

用程式物件操作資料庫。

工作原理：把 SQL 表轉成程式語言中的物件。

應用場景：Prisma、SQLAlchemy、Django ORM。

143. REST API｜REST 介面

以 HTTP 方法操作資源的 API。

工作原理：GET/POST/PUT/DELETE 對應讀寫刪改。

應用場景：教材、題庫、活動資料 API。

144. GraphQL｜圖形查詢語言

由 client 指定需要哪些資料。

工作原理：單一 endpoint，查詢結構由 query 決定。

應用場景：避免前端拿太多或太少資料。

145. Webhook｜事件回呼

某事件發生時自動通知外部系統。

工作原理：服務端把事件資料 POST 到指定網址。

應用場景：GitHub PR 更新後觸發自動檢查。

146. Rate Limit｜速率限制

限制一段時間內請求數。

工作原理：防止濫用、保護服務穩定與成本。

應用場景：AI 大量呼叫 API 時容易撞限制。

147. Pagination｜分頁

分批取得大量資料。

工作原理：用 page、cursor、limit 控制回傳範圍。

應用場景：AI 搜 GitHub issue 不能一次抓全部。

148. Idempotency｜冪等性

重複執行結果仍相同。

工作原理：同一請求重送不應造成重複副作用。

應用場景：AI 失敗重試寄信、付款、建立事件時很重要。

前端與互動介面

149. DOM｜文件物件模型

瀏覽器中的 HTML 結構樹。

工作原理：JavaScript 透過 DOM 讀寫畫面元素。

應用場景：AI 修一鍵複製、按鈕、搜尋互動。

150. Event Listener｜事件監聽器

等待使用者或系統事件。

工作原理：按下、輸入、滑動、載入時觸發函式。

應用場景：複製按鈕、篩選按鈕、表單送出。

151. RWD｜響應式網頁設計

不同螢幕都能良好閱讀。

工作原理：用 media query、彈性排版、相對尺寸。

應用場景：技術文章可在手機、平板、桌機閱讀。

152. Accessibility / A11y｜無障礙設計

讓更多使用者能操作網頁。

工作原理：考慮對比、鍵盤、標籤、語意 HTML。

應用場景：AI 產生 UI 後要檢查 aria-label 與標題結構。

153. Hydration｜水合

前端接手伺服器產生的 HTML。

工作原理：SSR 頁面載入後由 JS 綁定互動。

應用場景：Next.js / Nuxt 常見錯誤來源。

154. SSR｜伺服器端渲染

在伺服器產生 HTML。

工作原理：提升首屏速度與 SEO，但環境更複雜。

應用場景：內容網站、文件站、SEO 頁面。

155. CSR｜客戶端渲染

在瀏覽器用 JavaScript 產生畫面。

工作原理：初始 HTML 較少，互動性強。

應用場景：單頁應用、儀表板、互動教材。

156. SPA｜單頁應用

不重整頁面切換畫面的應用。

工作原理：路由在前端控制，資料透過 API 取得。

應用場景：互動式教材平台與後台管理系統。

三、AI Vibe Coding 建議流程

階段核心問題關鍵名詞產出物
1. 需求釐清到底要解決誰的什麼問題？User Story、Acceptance Criteria、Scope、MVPPRD、Spec、Backlog
2. 架構設計要用什麼技術、資料如何流動？Architecture、API、Schema、ADRDesign Doc、架構圖

最小修改
